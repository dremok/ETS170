Introduction

The purpose of this document is to serve as the main Requirements Document for an electronic voting system created during the course ETS170 at Lunds Tekniska Högskola, Sweden.

Swedish Valmyndigheten was founded in 2001 in order to plan and carry out nationwide referendums in Sweden. The small authority consisting of only 15 employees is supposed to operate as reliably and effectively as possible. During an election the workforce consists mainly of volunteers and Swedish Valmyndigheten is responsible for the premises and organization.

Now they want to give people the opportunity to vote from home and make the vote counting faster by digitizing parts of it. It should be easy, intuitive and safe to use. It is important that people can trust the system and that democracy is maintained while using it.

When the requirement specification is finished, Valmyndigheten will formulate a request for proposal and select a supplier through a tender process, as is common practice with government organisations.

At the end of the document, there is a glossary with definitions of terms used that might appear as vague or ambiguous.

Goals

The main goal of the project is to deliver a Requirements Document describing a safe electronic voting system that can be used during an election arranged by Swedish Valmyndigheten. It shall be possible to vote from home using an ordinary web browser and the voting results should be stored and counted at a central server.

Swedish Valmyndigheten has decided to introduce the electronic system to carry out the 2014 year parliamentary election and the business goals include reducing the amount of manual vote counting, paperwork and number of voting places.

Results from Elicitation

The following methods were chosen for elicitation:
  • Stakeholder analysis
    • The currently specified stakeholders were found through discussions in the group, and the interview with Thore Husfeldt contributed to the process of prioritizing them.
    • This elicitation method contributed especially to requirements regarding disabled voters and other voters with special needs for the user interfaces.
  • Interviewing
    • We interviewed expert Thore Husfeldt.
    • The interview led to the elicitation of requirements for reliable confirmation of votes, prevention of people selling their votes and prevention of coercion.
    • The interview also caused document studies of the Norwegian electronic election system to be carried out.
  • Interview with user
    • A typical voter was interviewed.
    • The interview revealed concerns for “hacker attacks” and risks of a general distrust of electronic systems.
    • Support for the idea of activating idlers was raised.
  • Document studies
    • We have studied specifications of the Norwegian election system, e-valg.
    • Principles of how different layers of encryption is applied in this system may be carried over to the system to be specified.
  • Prototyping
    • We have implemented a prototype of the web GUI which have been tested by the customer. The received feedback was valuable for specifying GUI requirements.

Results from Specification

So far, we have specified data requirements, functional requirements, design requirements and quality requirements for the system. The techniques we have used are task descriptions, a data dictionary, function requirements and a context diagram which shows an overview of the system domain.

Results from Prioritization

We have prioritized all of the stakeholders on a scale from 1 to 4. Then, for every stakeholder except the customer, we have estimated a priority for all system features. The customer prioritized the features themselves. Finally, the feature priorities were weighted against the stakeholder priorities. See the Release Plan for further clarification.

Results from Validation

We have written a checklist based on the one in the book "Software Engineering" by Soren Lauesen, sec. 9.4, and borrowing parts from "Framgangsrik kravhantering" by Joachim Karlsson, sec. 3.6. Each member of the group reviewed the entire requirements document according to the checklist and wrote down found errors. Then we had a review meeting where we compiled the errors and decided what to fix for version 2 of the document.

R02. The system shall support the following tasks:

Task T1: Carry out voting via the web interface

AttributesValues
Rationale: Let a person place a vote online.
Trigger: A voter opens up the voting website during the voting period.
Critical: Worst case: the entire voting population logs in and place their votes at the same time.

Sub-tasks:

Task T1.1 Authenticate: The voter enters his/her personal code on the received voting card in the web interface and authenticates him-/herself using electronic identification

AttributesValues
Example: By for example using BankID

Task T1.2 Choose party and candidate: The voter chooses the party and candidate that he/she wishes to vote for in the web interface.

Task T1.3 Confirm: The voter confirms that the chosen party and candidate is indeed the one he/she wants to vote for, using the web interface.

Task T1.4 Receive confirmation: The voter receives a personalized message confirming that the vote was counted for the correct party.

AttributesValues
Comment: It should be very difficult to fake this confirmation message.

Variants:

Task T1a Carry out voting via the web interface - the typical case: A person without any special needs uses the web interface to vote

AttributesValues
Example: The voter chooses to vote through the web interface. On the voting card there is a web adress to the voting website. The voter opens up this site in a web browser, logs in using electronic identification enters the personal code stated on the voting card. A party and a candidate is chosen in the web interface. The voter clicks on a 'Submit' button. After an appropriate amount of time, the voter receives some kind of personalized message confirming that the vote was counted for the correct party.

Task T1b Carry out voting via the web interface - visually impaired: A person with a visual impairment uses the web interface to vote

AttributesValues
Example: The voter chooses to vote through the web interface. Assuming that the voter has the necessary tools to surf the web without hinder, the voting should be carried out with the same ease as with the procedure in 1a for a fully sighted person.

Task T1c Carry out voting via the web interface - non-Swedish speaking voter: A person who doesn't understand Swedish uses the web interface to vote

AttributesValues
Example: The voter chooses to vote through the web interface. From the voting card it should be clear enough what is the web adress to the voting web site that no text on the card needs to be understood. As soon as the person enters the web site, there should be a clear possibility to change language to any language that is spoken by a Swedish citizen. The procedure is then carried out as in 1a.

Task T1d Carry out voting via the web interface - voter has motor impairments: A person with motor impairments uses the web interface to vote

AttributesValues
Example: The voter chooses to vote through the web interface. Assuming that the voter has the necessary tools to surf the web without hinder, the voting should be carried out with the same ease as with the procedure in 1a for a person without motor impairments.

Task T2: Carry out voting via machine

AttributesValues
Rationale: Let a person place a vote on a machine.
Trigger: A voter goes to a place with a voting machine and wants to place a vote.
Frequency: The machine might be used continuously during open hours throughout the entire voting period.

Sub-tasks:

Task T2.1 Authenticate: The voter shows an accepted identification document and scans his or her identification card.

Task T2.2 Choose party and candidate: The voter chooses the party and candidate that he/she wishes to vote for in the web interface on the machine.

Task T2.3 Confirm: The voter confirms that the chosen party is indeed the one he/she wants to vote for, using the web interface on the machine.

Task T2.4 Receive confirmation: The voter receives a personalized message confirming that the vote was counted for the correct party.

AttributesValues
Comment: It should be very difficult to fake this confirmation message.

Variants:

Task T2a Carry out voting via machine - the typical case: A person without any special needs votes using a machine at a voting place

AttributesValues
Specification: The voter uses a computer which is designed to only operate the web voting interface. The voter shows his/her valid identification to at least two election inspectors and scans some kind of barcode (or similar) on the personal voting card. The voter is assigned a voting booth where a computer is situated. This computer has minimal functionality and can only operate the web interface for voting. Confirmation message is sent to the voter in the same way as in Task 1.

Task T2b Carry out voting via machine - visually impaired: A person with a visual impairment votes using a machine at a voting place

AttributesValues
Specification: The public voting machines are equipped with technologies that enable a visually impaired person to use them. This makes voting identical to the case in 1b.
Example: The voting machines might be placed in sound isolated booths or be equipped with headphones in order to provide text-to-speech functionality.
Comment: In case text-to-speech is used, it should not be possible to hear the output from outside of the booth where the machine is placed.

Task T2c Carry out voting via machine - non-Swedish speaking voter: A person who doesn't understand Swedish votes using a machine at a voting place

AttributesValues
Specification: This should be identical to 1c.

Task T2d Carry out voting via machine - voter has motor impairments: A person with motor impairments votes using a machine at a voting place

AttributesValues
Specification: The public voting machines are equipped with voice recognition functions to enable the voter to use them. The general procedure is identical to the typical case (Task T2a).
Comment: It must not be possible to conclude what was voted for from outside the voting booth. One example is to use sound isolated booths. Another way might be to construct the user interface in a way that does not encourage the voter to say out loud what (s)he is voting for.

Task T2e Carry out voting via machine - voter has cognitive impairments: A person with cognitive impairments votes using a machine at a voting place

AttributesValues
Specification: The designers of the user interface should strive to make it as easy and efficient to use as possible.

Data Requirements

R03. The server shall handle the following data:

Data dictionary

Class Party Candidate: A member of a party who the voters can vote for.

AttributesValues
Specification: A party candidate is party appointed candidate. A Party Candidate is eligible to receive candidate votes. Also knows which party it belongs to.
Example: (1) Anna Troberg of the pirate party. (2) "blank" of any party.
Comment: It should be possible to create, read, update and delete party candidates. CRUD is only applied during the registration phase; during the other phases it is only possible to read party candidates.

Class Party: A party which can receive a vote from the voters.

AttributesValues
Specification: Each votable party is represented with a party in the system, there must also be a "no-vote" party in order to mask the fact that some people may not have voted, or may have voted blank.
Example: (1) The "no-vote" party. (2) The pirate party.
Comment: It should be possible to create, read, update and delete parties. CRUD is only applied during the registration phase; during the other phases it is only possible to read parties.

Class Vote: A vote can be placed by a voter on a party and candidate.

AttributesValues
Specification: A Vote is placed by a voter on a party and a party candidate. The vote is masked in such a way that there is no way for an outsider to determine which candidate and party the vote was placed on, nor who placed the vote, while still being connected to its voter. The party candidate which is being voted upon must be a party candidate from the party being voted on.
Example: (1) A vote on pirate party and "blank" candidate. (2) A vote on "blank" party and "blank" candidate
Comment: It should only be possible to create and read votes, not delete or update them. Votes can only be created during the voting phase and read during the tallying phase.

Class Voter: Can vote in the election

AttributesValues
Specification: A voter is someone who has voting rights in Sweden. The point of the voter class is to make sure one individual may only have one of its respective votes counted. A voter must at all times have a vote associated with it. The model must be able to hide whether a voter has voted or not as well as which party it has voted upon. A voter must for each votable party have a personal verification code. Every voter must have a means to authenticate him- or herself via the authentication system in use.
Example: (1) A person who has voting rights but does not vote. (2) A person who has voting rights and does place a vote.
Comment: It should be possible to create, read, update and delete voters. CRUD is only applied during the registration phase; during the other phases it is only possible to read voters.

Glossary

Below are the definitions of a number of terms used throughout the requirements document.

Coercion:

The phenomenon where a voter is somehow persuaded by someone else to place a vote against their will. This includes both being forced to place a vote at all, and being forced to vote for a party and/or a candidate one does not wish to vote for.

Voting machine:

This refers to machines placed in a public voting place, i.e. not to personal computers in voters’ homes.

Web interface:

The web interface is present both in the web application accessible to any personal computer and in the voting machines at the designated voting places. However, there are small differences between the two environments, such as in the authentication method used.

Paper vote:

A vote placed on paper, i.e. through mail or a ballot paper.

Electronic vote:

A vote placed through the web interface, either from a public voting place or from a personal computer.
